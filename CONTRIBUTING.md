# Contributing

Thanks for taking the time to contribute to a VeriFast project. This is the organization-wide contributing
guide. Individual repositories may add their own `CONTRIBUTING.md` with project-specific details, which takes
precedence over this one.

## Getting started

1. Fork or clone the repository.
2. Create a branch off the default branch for your change.
3. Make your change, keeping commits focused and self-contained.
4. Open a pull request against the default branch.

## Branching

- Branch names should be short and descriptive, prefixed by type: `feature/`, `fix/`, `chore/`, or `docs/`.
- Keep one logical change per branch where possible.

## Commits

- Write clear, present-tense commit messages that explain the intent of the change, not just the mechanics.
- Group related work into coherent commits rather than one large catch-all commit.

## Pull requests

- Fill out the pull request template.
- Keep pull requests focused; smaller, reviewable changes are merged faster.
- Make sure any automated checks pass before requesting review.
- Link the relevant issue where one exists.

## Code style

- Follow the conventions already present in the repository you are working in.
- Where a repository defines linting or formatting tooling, run it before pushing.

## Reporting issues

Use the issue templates to report bugs or request features. For security vulnerabilities, do **not** open a
public issue — follow the process in [`SECURITY.md`](SECURITY.md).

## Code of conduct

By participating in a VeriFast project, you agree to uphold our [Code of Conduct](CODE_OF_CONDUCT.md).
